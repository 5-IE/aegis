Aegis API Reference

Attendance backend for Aegis. All application endpoints listed below are proposed. Auth endpoints (/auth/*) are already implemented.

Introduction

The Aegis backend serves two first-party native clients: an iPhone learner app and a macOS admin app. Learners ping the backend every N minutes when their phone detects a classroom iBeacon; admins view live occupancy, attendance history, and configuration. All responses are JSON.

Base URL

https://aegis.example.com

Paths in this document are shown relative to that base. Auth endpoints use /auth/*; application endpoints are versioned under /api/v1/*.

Conventions

  • Field naming: request and response bodies use snake_case.
  • Timestamps: ISO-8601 UTC, e.g. 2026-07-03T08:14:22Z, unless otherwise stated.
  • Dates: YYYY-MM-DD, interpreted in the deployment timezone from SYSTEM_CONFIG.timezone.
  • Times of day: HH:MM:SS, 24-hour, in the deployment timezone.
  • Content type: requests with bodies send Content-Type: application/json.
  • Body size: the server rejects request bodies larger than 64 KB.

Authentication

All application endpoints require an access token in the Authorization header. Tokens are obtained via POST /auth/login and refreshed via POST /auth/refresh. See the Auth section below for the full flow.

Authorization: Bearer <access_token>
BadgeMeaning
publicNo authentication required.
learnerRequires a learner access token.
adminRequires an admin access token.

Error format

All error responses share the same shape:

{
  "error": "invalid_credentials",
  "message": "Username or password is incorrect"
}
CodeStatusMeaning
invalid_request400Body malformed or fails schema validation.
invalid_credentials401Login credentials rejected.
invalid_grant401Refresh token unknown, expired, or reused.
unauthorized401Missing or invalid access token.
forbidden403Authenticated, but role is insufficient.
not_found404Resource does not exist.
too_many_requests429Rate limit exceeded.
internal_error500Unexpected server error.

Log in implemented

POST /auth/login public

Exchange username and password for an access token and rotating refresh token.

Request body

FieldTypeConstraints
usernamestring1–50 chars
passwordstring1–72 chars

Response 200

{
  "access_token": "eyJhbGciOi...",
  "refresh_token": "5f4b7c...base64url",
  "expires_in": 900,
  "user": {
    "id": 42,
    "username": "alice",
    "role": "learner",
    "session": "AM",
    "first_name": "Alice",
    "last_name": "Doe",
    "email": "alice@example.com"
  }
}

Errors

  • 400 invalid_request — body malformed.
  • 401 invalid_credentials — user unknown or wrong password.
  • 429 too_many_requests — 5 failures per (ip+username) in 15 min.

Refresh access token implemented

POST /auth/refresh public

Rotate a refresh token; returns a new access + refresh pair. Reusing an already-revoked refresh token cascade-revokes every refresh token for that user.

Request body

FieldTypeConstraints
refresh_tokenstring1–256 chars

Response 200

{
  "access_token": "eyJhbGciOi...",
  "refresh_token": "9c1a2f...base64url",
  "expires_in": 900
}

Errors

  • 400 invalid_request
  • 401 invalid_grant — token unknown / expired / reused.
  • 429 too_many_requests

Log out implemented

POST /auth/logout public

Revoke a refresh token. Idempotent — an unknown or already-revoked token still returns 204. The paired access token remains valid until its 15-minute expiry.

Request body

FieldTypeConstraints
refresh_tokenstring1–256 chars

Response

  • 204 No Content

Return the authenticated learner's profile.

Response 200

{
  "id": 42,
  "first_name": "Alice",
  "last_name": "Doe",
  "username": "alice",
  "email": "alice@example.com",
  "role": "learner",

The UI wireframe only reads first_name; the additional fields are returned as a convenience so the app doesn't need to re-fetch after login.

Learner dashboard

GET /api/v1/dashboard learner

Aggregated attendance counters and today's status.

Response 200

{
  "total_attendance": 92,    // ATTENDANCE_HISTORY rows where status in ('early','late')
  "total_late": 7,           // ATTENDANCE_HISTORY rows where status = 'late'
  "leave_taken": 3,          // ATTENDANCE_HISTORY rows where status = 'leave'
  "today_status": "Checked In"
}

today_status values

ValueCondition
"Not Checked In"No presence log today, current time within or before session window.
"Running Late"No ping yet, current time past late_after, before end_time.
"Checked In"Ping received today; most recent ping within staleness window.
"Checked Out"Ping received today; session end_time has passed and last ping is older than the staleness window.
"Not Checked Out"Session ended, but a ping arrived within the staleness window (still detected).
"Off"Today's ATTENDANCE_HISTORY row is leave, or the user has no active session assignment.

Attendance history

GET /api/v1/histories learner

Paged list of the learner's daily attendance rows. Times of day are derived from the first and last PRESENCE_LOG ping on each date.

Query parameters

ParamTypeNotes
monthinteger 1–12Optional filter; requires year.
yearintegerOptional filter.
pageintegerDefault 1.
per_pageintegerDefault 20, max 100.

Response 200

{
  "list": [
    {
      "date": "2026-07-01",
      "checked_in_at": "2026-07-01T08:04:12Z",
      "checked_out_at": "2026-07-01T11:58:41Z",
      "status": "early"
    },
    {
      "date": "2026-07-02",
      "checked_in_at": null,
      "checked_out_at": null,
      "status": "leave"
    }
  ],
  "page": 1,
  "per_page": 20,
  "total": 137
}

status values

  • "early" — first ping before SESSION_CONFIG.late_after.
  • "late" — first ping at or after late_after, before end_time.
  • "leave" — recorded leave.
  • "absent" — no ping in the session window.

Report presence

POST /api/v1/presence learner

Called by the iPhone app whenever it detects an iBeacon. The backend resolves the beacon to a room and writes a PRESENCE_LOG row.

Not in the original UI list, but the read endpoints have nothing to serve without it. This is the write path that populates PRESENCE_LOG.

Request body

FieldTypeNotes
beacon_identifierstring1–100 chars, matches DEVICE.identifier.
position_xfloatOptional; forwarded to PRESENCE_LOG.position_x.
position_yfloatOptional.
battery_levelinteger 0–100Optional; phone battery %.

Response

  • 204 No Content

Errors

  • 400 invalid_request — body malformed, or beacon unknown / unassigned to a room.
  • 429 too_many_requests — hard cap 20 pings per learner per minute.

Room live map

GET /api/v1/admin/rooms/:room_id/map admin

Latest known position of each user currently in the room, based on the most recent PRESENCE_LOG per user within the staleness window (SYSTEM_CONFIG.presence_staleness_minutes).

Response 200

{
  "list": [
    {
      "id": 10921,
      "user": { "id": 42, "name": "Alice Doe", "session": "AM" },
      "x": 4.2,
      "y": 7.1
    }
  ]
}

Errors

  • 404 not_found — room does not exist.

Current occupants

GET /api/v1/admin/rooms/:room_id/current-occupants admin

Users currently in the room (last ping in the room within the staleness window). duration_seconds is the elapsed time since the user's first ping today, regardless of room switches.

Response 200

{
  "list": [
    {
      "user": { "id": 42, "name": "Alice Doe", "session": "AM" },
      "duration_seconds": 4823,
      "status": "Checked In"
    }
  ]
}

Errors

  • 404 not_found — room does not exist.

Room additional data mocked

GET /api/v1/admin/rooms/:room_id/additional-data admin

Environmental readings for the room. Mocked with fixed values in v1 — no sensor table yet.

Response 200

{
  "room_temperature": 24.5,   // celsius
  "humidity": 62,               // percent
  "people_in_room": 18          // current occupant count
}
Temperature and humidity return a fixed value in v1. people_in_room uses the same live count as the occupants endpoint. Real sensor integration is a future milestone.

Partial update. Only fields present in the body are changed. Effective within 30 seconds across the fleet (in-process TTL cache).

Request body

FieldTypeNotes

Errors

Appendix

Rate limits

  • POST /auth/login: 5 failed requests per (ip + username) per 15 min.
  • POST /auth/refresh: 5 failed requests per ip per 15 min.
  • POST /api/v1/presence: 20 requests per learner per minute.

Nightly rollup

A scheduled script (npm run rollup) writes yesterday's ATTENDANCE_HISTORY row per learner from PRESENCE_LOG + SESSION_CONFIG. Existing leave rows are preserved. The job is idempotent.

Out of scope for v1

  • Explicit user-triggered check-in / check-out.
  • Leave request and approval flow.
  • Real IoT sensor readings.
  • Multi-timezone operation.
  • Real-time push (all endpoints polled).
  • Back-fill of historical attendance for pre-existing data.
  • Anti-spoofing of presence pings.
